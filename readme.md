System OverviewThis system is an automated Crypto Intelligence Pipeline hosted on an AWS EC2 instance (Ubuntu). It listens to "Call Channels" (sources of alpha), extracts token contract addresses, enriches them with real-time market data (DexScreener), forwards them to a central "Destination Channel" (organized by Topics), and tracks their performance over time to generate profitability reports.Core Componentsforwarder.py (The Listener): Real-time message interceptor and enricher.watcher.py (The Tracker): Background process that monitors token prices and updates the database.admin_bot.py (The Controller): A Telegram-based command interface for managing the server.reporter.py (The Analyst): Generates ad-hoc or scheduled performance reports.crypto_data.db: SQLite database storing all calls, prices, and performance metrics.2. Database Schema (crypto_data.db)The system uses SQLite with two primary tables.Table 1: calls (Primary Signal Log)Stores the first time a token is mentioned by a channel.id (Integer, Primary Key): Unique ID.timestamp (String): ISO format (YYYY-MM-DD HH:MM:SS).channel_id (Integer): ID of the source Telegram channel.channel_name (Text): Name of the source channel.symbol (Text): Token ticker (e.g., "PEPE").ca (Text): Contract Address (Unique Identifier).entry_price (Float): Price at the moment of the call.entry_mcap (Float): Market Cap at entry.entry_vol_5m (Float): 5-minute volume at entry.initial_liquidity (Float): Liquidity at entry.max_price (Float): Highest price reached since tracking began.current_price (Float): Latest fetched price.peak_multiplier (Float): Calculated as max_price / entry_price.is_rug (Boolean/Int): 1 if Liquidity < $500, else 0.is_active (Boolean/Int): 1 if currently being tracked, 0 if expired/rugged.Table 2: re_entries (Secondary Log)Stores subsequent mentions of the same token by the same channel after the cooldown period.Schema is identical to calls table.3. Component Logic & WorkflowsA. forwarder.py (The Listener)Role: Monitors Telegram channels, parses content, fetches data, and forwards cleaned messages.Status: Runs as a Systemd Service (cryptobot).Logic Flow:Startup: Loads config.json (Source IDs $\rightarrow$ Destination Topic IDs).Event Loop: Listens for NewMessage events in configured source channels.Parsing:Extracts Contract Addresses (CA) using Regex (Solana/EVM patterns).If NO CA found: Copies the message text/media to the destination topic (Preserves non-token alpha).If CA found: Proceed to Step 4.Data Enrichment (DexScreener API):Fetches Price, Market Cap, Liquidity, Volume.Fetches Socials (Twitter/Telegram links) and Banner/Logo images.Database Logging:Spam Check: If the same channel called the same CA < 2 hours ago $\rightarrow$ IGNORE.Re-entry Check: If called > 2 hours ago $\rightarrow$ Log to re_entries table.New Call: If never called $\rightarrow$ Log to calls table.Activation: Sets is_active = 1.Message Construction:Formats a "Call Card" with: Channel Name, Token Symbol, Entry Price/MC/Liq.Context: Fetches reply message text (if the call was a reply) to provide context.Links: Generates clickable links for Maestro, DexScreener, and Socials.Transmission:Sends the message to DESTINATION_CHAT_ID inside the specific TOPIC_ID.Media: Uses the Token Banner (if available) or Logo; falls back to original message media.B. watcher.py (The Tracker)Role: Updates prices and detects scams/rugs for active tokens.Status: Runs as a parallel thread/process (usually invoked via forwarder.py or separate service).Logic Flow (Smart Triage):Selection: Reads all rows where is_active = 1.Prioritization (Triage):Young (< 1 hour): Checked every 1 minute.Mid-Life (1-24 hours): Checked every 5 minutes.Old (> 24 hours): Checked every 30 minutes.Batching: Groups tokens into batches of 30 CAs to respect API rate limits.Update Logic:Fetches current price/liquidity from DexScreener.ATH Update: IF current_price > max_price THEN max_price = current_price.Multiplier: Recalculates max_price / entry_price.Kill Switches (Stop Tracking):Rug Pull: IF liquidity < $500 $\rightarrow$ Set is_rug=1, is_active=0.Expiry: IF timestamp > 7 days $\rightarrow$ Set is_active=0.C. admin_bot.py (The Manager)Role: Allows the owner to manage the bot via Telegram DM or Group.Status: Runs as a separate Systemd Service (adminbot).Key Features:Ghost Mode: If a command is sent in a public channel, the bot deletes the user's command and its own reply after 5 seconds to keep the chat clean.Security: Only responds to ADMIN_ID.Commands:/add <channel_id> <topic_id>: Adds a new mapping to config.json and auto-restarts the Forwarder./restart: Forces a systemctl restart cryptobot./report: Safely pauses the Forwarder (releases DB lock), runs reporter.py, and resumes the Forwarder./settings: Displays current configuration.D. reporter.py (The Analyst)Role: Generates visual performance reports.Status: Triggered manually via /report or Cron Job.Logic Flow:Query: Selects calls from the last 7 days.Processing:Trimming: Trims channel names to 10 chars (stripping whitespace).Metrics: Calculates Win% (>2x multiplier), Rug% (Liquidity < $500), and Max Multiplier.Formatting:Constructs a Monospace Table manually (using f-strings) for perfect alignment on mobile.Delivery: Sends the report to the General topic of the destination channel.4.
